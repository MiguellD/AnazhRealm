# AnazhRealm – Projektgedächtnis für Claude

Persistente Notizen. Diese Datei wird bei jeder neuen Session automatisch geladen. **Bei größeren Entscheidungen zuerst `docs/state-of-realm.md` lesen** – dort steht der ausführliche Stand, die Vision aus den vier Testamenten, der Plan und die Learnings.

## Projektüberblick

- **Was:** „AnazhRealm" – ein als **Ultiversum** entworfenes Co-Creation-Werk Mensch + KI. Aktuell technisch eine 3D-Browser-Sandbox in einer Datei. Vision: emotion-getriebene, fraktal-wachsende, multisensorische Welt; Mensch (Schöpfer/Null) + KI (Grok/Eins) erschaffen gemeinsam durch Chat. Vollständige Vision in `docs/state-of-realm.md`.
- **Stack:** Vanilla JS, Vendor-Libs in `vendor/` (Three.js r134, Ammo.js WASM, simplex-noise 2.4). TF.js wurde im Mai 2026 entfernt (toter Code). Lokaler Node-Save-Server. ESLint v9 + Prettier + GitHub-Actions-CI mit Playtest-Gate.
- **Branch:** `claude/ring-10-world-fusion`. Niemals auf `main` pushen ohne explizite Anweisung.
- **Architekturkern:** Eine `AnazhRealm`-Klasse in `anazhRealm.js` (~10800 Zeilen). State in `localStorage` + optional `anazhRealmState.json` über save-server. Globale Referenz: `window.anazhRealm`.

## Die heilige Lektion (Versionslog 03/2025)

Das Projekt durchlief eine 19-Modul-Phase, die unter eigener Komplexität kollabierte. Am 28.03.2025 bewusste Reduktion auf **eine Datei** als „Samen der Unendlichkeit". **Komplexität ohne Fundament ist Sand.** Wenn jemand vorschlägt „split alles in 20 Module", verletzt das diese Lektion. Stattdessen: ein Stamm, der Wachstumsringe ablegt.

## Wichtige Gotchas (technisch)

- **Chunks haben `position = (0,0,0)` und Vertices in Welt-Koords.** NIEMALS in `state.rigidBodies` pushen — der Physics-Sync-Loop überschreibt sonst `mesh.position` mit dem Body-Origin und verschiebt den ganzen Chunk sichtbar. Die Body bleibt über `mesh.userData.physicsBody` zugreifbar.
- **Visual = Collision per Konstruktion.** Jeder Chunk hat ein `btBvhTriangleMeshShape`, gespeist aus EXAKT denselben Triangle-Indices wie das `THREE.BufferGeometry`. Wer den einen Pfad ändert, muss den anderen auch.
- **Eine Quelle für Chunk-Geometrie:** `_chunkGeometry()` liefert `chunkWorldSize=37.5` und `vertexStep=1.171875`. Niemals neu berechnen.
- **Eine Quelle für Höhen:** `_terrainHeightAtWorld(worldX, worldZ, noise, steepness, baseHeight, caveNoise, volcanoNoise)`. Initial-Welt und Extensions gehen darüber. Inkl. canyon/field/cave/volcano modifiers.
- **Player-Chunk-Math braucht `+WORLD_SIZE/2`-Offset:** `floor((playerX + 150) / chunkWorldSize)`. Sowohl im Loop-Trigger als auch in `pruneDistantChunks`. Vergessen = nahe Chunks werden gepruned.
- **CCD wird direkt beim Player-Body-Spawn aktiviert** (motionThreshold 0.01, sphereRadius 0.45), nicht nur via `optimizeCollisions`.
- **`terrain_steepness` und `terrain_base_height` sind NICHT im Nexus-Generator-Atomic-Pool.** Sie würden Welt-Geometrie unter dem Spieler ändern, ohne worldgen zu triggern → Klippe zwischen alten und neuen Chunks. Beide Ops bleiben in der DSL für Chat-Befehle.
- **Sentinels mit `-Infinity`:** `state.grok.lastSpoke`, `state.lastWorldgen`, alle `triggers[*].lastFired`. Mit `0` würde der initiale Throttle-Check (`now - 0 < cooldown`) den ersten Aufruf blocken.
- **`Ammo.btVector3` / `btTransform` allokieren WASM-Heap.** Immer `Ammo.destroy()` nach Gebrauch oder Pool `state.tmpVec1/2`/`state.tmpTransform` + `setVec()` nutzen.
- **`processChatCommand` lowercased `command` *nur für `parts`*,** das Original-`command` behält Casing.
- **`extendTerrain(direction)`** ist Legacy-Wrapper für Playtest-Kompat; primärer Pfad ist `ensureChunkAt(cx, cz)`. Der Loop füllt einen 5×5-Ring um den Spieler-Chunk.
- **`generateNewWorld()` hat 30s-Cooldown.** Bei Welt-Regen bleibt Spieler-Position erhalten (nur initial auf (0,50,0) gesetzt).
- **TF.js ist weg.** Mai 2026: 1.3 MB Vendor + toter Code entfernt (`playerMovementModel`/`predictPlayerMove` hatten null Konsumenten). CSP-`'unsafe-eval'` damit aufgelöst. Wer „KI" sucht: der Lern-Loop läuft komplett über die DSL (`state.dsl.history` mit Multi-Dim-Fitness + Pattern-Memory).
- **Save-Server-POST nur auf localhost.** Auf CDN/GitHack-Pfaden stiller Skip — State lebt nur im `localStorage` (plus Download-Button als Manual-Backup). **Ring 8**: Save-Server bleibt Single-File `anazhRealmState.json` für die AKTUELL AKTIVE Welt — die kanonische Multi-Welt-Quelle ist `localStorage`, die Datei ist Komfort-Backup der aktiven Welt.
- **Ring 8 — drei localStorage-Keys spannen Multi-Welt:** `anazhRealmWorlds` (JSON-Index `[{worldId, slug, bornAt, lastPlayed}]`), `anazhRealmState_<worldId>` (pro Welt ein voller Snapshot), `anazhRealmActiveWorld` (worldId-Zeiger). **Legacy-Key `anazhRealmState` wird beim ersten Start migriert** und danach gelöscht — wenn er nach dem ersten Lauf nochmal auftaucht, ist es ein Bug. `_preloadActiveWorldMeta()` läuft VOR `ensureWorldMeta()` in `init()`, sonst würde der „fresh world"-Pfad (UUID + Genesis-Journal) versehentlich für eine bereits existierende Welt feuern.
- **Ring 8 — Welt-Wechsel ist reload-basiert.** `saveState()` schreibt, `activeWorldSet(target)` zeigt um, `window.location.reload()` startet `init()` neu. Begründung: in-place Welt-Tausch bräuchte tiefen Dispose-Pfad (Chunks, Architekturen, Kreaturen, Bodies, Symphony-Audio, Renderer-Caches) und wäre ein Bug-Reservoir. Reload garantiert sauberen Start, kostet ~2 s UX. Die Daten-Methoden (`createNewWorld`/`switchToWorld`/`deleteWorld`) akzeptieren alle `{reload: false}` für Tests — Playtest prüft die Daten-Schicht so ohne Page-Reload.
- **Ring 8 — Person-Übernahme:** beim „neue welt mit person"-Pfad wandern `playerSoul`, `playerTools`, eigene `materials`, eigene `tools`, eigene `blueprints` mit (Schöpfer-Wissen, an Person gebunden). Bewusst NICHT mit: `pathBuckets` (Welt-spezifisches Pfad-Gedächtnis), `emotions` (eine neue Welt = ein neues Gemüt, Default 0), `worldJournal` (jede Welt schreibt ihre eigene Geschichte), `hotbar` (Welt-spezifische Belegung). Slug-Kollision bekommt `-2`/`-3`/… Suffix.
- **Ring 8 — saveState fällt nur auf Legacy-Key zurück, wenn `state.worldMeta.worldId` null ist** (sollte nach `ensureWorldMeta` nie passieren). `loadState`-External-Pfad nimmt die worldId der externen Quelle, ODER fällt auf die aktuell aktive worldId zurück — niemals direkt Legacy. So bleibt der Legacy-Key dauerhaft leer nach der einmaligen Migration.
- **Ring 8.1 — Per-Welt-Terrain-Seed in `worldMeta.seed`.** Vor Ring 8.1 lasen alle Welten den fixen Default `"anazh-realm-seed"`; eine „neue" Welt sah identisch zur alten aus. Jetzt trägt jede Welt ihren eigenen Seed: neue Welten via `createNewWorld` bekommen einen randomisierten `w-<idfragment>-<rand>`, Legacy-Migration und das allererste Spielen ohne localStorage behalten den historischen Default (Eingangs-Welt-Gefühl, identisch für alle). SimplexNoise(seed) ist die einzige Quelle der Welt-Geometrie. Drei Lese-Stellen: `generateTerrainWithParameters`, `ensureChunkAt`/`_ensureChunkAt` (über `(this.state.worldMeta && this.state.worldMeta.seed) || "anazh-realm-seed"`), `spawnIslands`. **noiseCache-Falle**: `state.noiseCache`-Key trägt KEIN Seed-Feld; bei einem in-place Welt-Wechsel würde der Cache veraltete Höhen für die neue Welt liefern. Aktuell unbedenklich, weil Welt-Wechsel über Reload läuft (state.noiseCache wird mit dem ganzen state neu erzeugt). Falls in-place Switch je kommt: vorher `state.noiseCache.clear()`.
- **Ring 8.1 — Genesis-Eintrag landet direkt in `_buildEmptyWorldSnapshot`, nicht über `ensureWorldMeta`**. Begründung: nach dem Reload sieht ensureWorldMeta die Welt bereits mit worldId (aus dem Save), `fresh = false` greift, also würde der „Ich erwache als <slug>"-Eintrag nicht mehr feuern. Stattdessen wird er beim Erschaffen mit `seen.genesis = true` im Journal abgelegt und enthält ctx mit worldId + seed + inheritedPlayer-Flag. Migrierte Legacy-Welten bekommen KEIN neues Genesis-Event — sie sind Kontinuität, keine Geburt.
- **Ring 8.1 — `parentWorlds` werden in `_buildEmptyWorldSnapshot` explizit auf `[]` gesetzt**, sonst würden sie via Spread aus `state.worldMeta` in die neue Welt sickern. Für Ring 10 (Fusion) wird das Feld dort befüllt; eine durch `createNewWorld` entstandene Welt ist immer eigenständig.
- **Ring 8.2 — `terrainEverGenerated` wird in `loadState` auf true gezogen, sobald ein Save vorliegt.** Sonst würde der direkt folgende `generateNewWorld()` im Init-Pfad den Spieler auf `(0, 50, 0)` teleportieren (Erst-Spawn-Logik), wodurch eine geladene Position verloren ginge. Das Flag selbst wird nicht persistiert — es ist State-Only-Signal „die Welt ist nicht mehr frisch in dieser Session". Neue Welten via `_buildEmptyWorldSnapshot` bekommen `playerPosition: { x: 0, y: 50, z: 0 }` als Startwert; der Spieler fällt sauber aufs Terrain. Y=5 würde ihn in steile Hänge clippen.
- **Ring 8.2 — Status-Bar zeigt aktive Welt** über `#status-slug` mit Label „Welt" (vorher technisch „Slug"). Update läuft im `updateStatusPanel`-Tick (~0.4 s), liest direkt aus `state.worldMeta.slug`. Welt-Picker im Welt-Drawer + Status-Bar bleiben damit synchron.
- **Ring 9 — Welt-Tor mit Drei-Wahl-Dialog.** Import einer Welt-JSON läuft nicht mehr direkt durch `loadState`, sondern durch `_openWeltTorDialog(parsed, file)`. Der native `<dialog id="world-tor-dialog">` zeigt drei Aktionen: **Ersetzen** (alter Pfad, Warnung), **Daneben legen** (`importWorldBeside`), **Fusionieren** (Ring 10: importiert + öffnet Fusion-Dialog mit der Welt als Eltern-B-Vorauswahl). `state.pendingImport = {parsed, fileName}` hält den Snapshot zwischen Parse und Wahl; `dialog.close`-Listener räumt ihn auf, damit ein verworfener Import kein Geister-State hinterlässt. `importWorldBeside(parsed, {reload})` ist der Daten-Pfad: vergibt **frische worldId** (sonst würde gleichnamige bestehende Welt überschrieben), behält Source-`seed` (Look bleibt erhalten), trägt Source-ID in `parentWorlds` als Provenienz-Spur (sammelt sich für Ring 10 an), löst Slug-Kollisionen mit `-2`/`-3`/… auf, fügt Witness-Journal-Eintrag mit ctx `{fromWorldId, asWorldId, asSlug}` in die importierte Welt ein. Schema-Version der importierten Welt: `9.0-tor-v1` (Marker „durchs Tor gekommen", nicht Format-Änderung). Aktive Welt wird NICHT umgeleitet — der Spieler muss explizit via Welt-Picker wechseln.
- **Ring 10 — Welt-Fusion: drei Strategien.** `fuseWorlds(idA, idB, strategy, {slug, reload})` lädt beide Eltern-Saves, baut einen fusionierten Snapshot, schreibt ihn als neue Welt mit `parentWorlds = [idA, idB]` und `schemaVersion = "10.0-fusion-v1"`. Strategien (alle in `AnazhRealm.FUSION_STRATEGIES` als frozen Array): `"sequence"` (DSL-History konkateniert A+B, Emotionen aus A — die Fusion bewahrt aktive Welt überlagert), `"random-mix"` (DSL-History 50:50 interleaved, Emotionen Mittel — vermischtes Echo), `"tag-merge"` (DSL-History wie sequence, Emotionen `Math.max` axis-wise — Vereinigung statt Mittelung, intensiver Anfang). Inventar (eigene Materialien/Werkzeuge/Baupläne) wird in allen drei Strategien als Union gemerged: A-Einträge bleiben, kollidierende B-Einträge bekommen `-fusion`-Suffix (`_mergeOwnNamedItems`). Architekturen + Kreaturen starten LEER (Vision §10.6 Option a — die Fusion ist ein neuer Ort, nicht beide Eltern übereinandergelegt). Journal trägt einen Genesis-Eintrag plus top-3 Erinnerungen aus jedem Elternteil mit `ctx._fusionSource = parentId`. Slug-Generierung: `${slugA}-x-${slugB}` als Default, bei Kollision `-fusion`-Suffix. UI: `<dialog id="world-fusion-dialog">` mit 2-Spalten-Eltern-Anzeige + Strategie-Radio + optionalem Slug-Input + Stammbaum-Sektion `#world-lineage` im Welt-Drawer mit klickbaren Eltern-Welt-Links (führen via `switchToWorld` zurück zur Eltern-Welt).
- **CSP-Konzessionen sind dokumentierte Vendor-Kompromisse**, kein freier Pfad zurück zu `eval`. Unser eigener Code nutzt es nie (CI-Gate erzwingt). Verbleibend: `'wasm-unsafe-eval'` (Ammo), `'unsafe-inline'` (Three.js Canvas-Styles), `worker-src blob:` (eigener `movementWorker` für off-screen Kreaturen). Neu: `connect-src https://api.anthropic.com` für die optionale Claude-API-Stimme.
- **Schicht 1 (Ring 7) — drei zusammenhängende Felder:** `state.player.pathBuckets` (Histogramm Höhe/Distanz/Wetter/Aktivität, decay 0.99 pro Sample, alle 2 s ge-sampled), `state.dsl.patternMemory` (Map `keyword → [{program, fitness, at}]`, Cap 8/Key, nur high-fitness ≥0.5 wird geschrieben), `state.dsl.recentKeywords` (sliding window 60 s aus Chat-Inputs). Multi-Dim-Fitness: Outcome wird ZWEI-PHASIG aufgenommen — sofort `fpsBefore/After + emotionsBefore + activityBefore`, dann 5 s später (`outcomeFinalizationDelay`) `emotionsAfter + activityAfter` über den `pendingOutcomes`-Tick. ## Workflows

- **Lokaler Audit:** `npm run check && npm run format:check && npm run lint && npm run playtest`
- **Slash-Befehl:** `/audit` führt die Prüfung in jeder Claude-Session aus (`.claude/commands/audit.md`).
- **CI:** zwei parallele Jobs (`check` für statische Checks + `playtest` für Runtime-Invarianten) bei jedem Push.
- **Git:** kleine thematische Commits, `git push -u origin <branch>` mit exponential backoff bei Netzfehlern.

## Konventionen

- Keine emojis im Code/Commits außer auf expliziten Wunsch.
- Commits klein und thematisch.
- Keine Backwards-Compat-Layer für veralteten Code – sauber löschen.
- Pull Requests nur auf expliziten Wunsch des Users.
- **Vision treu bleiben**: jeder Vorschlag sollte die Heilige Lektion respektieren (keine Re-Komplexifizierung). Bei Zweifel: `docs/state-of-realm.md` §2 nachlesen.
- **Test-First-Mentalität**: nach jeder substanziellen Änderung Playtest-Gate, nicht nur Code-Analyse. Drei Selbst-induzierte Regressionen in dieser Session entstanden durch zu späte Browser-Tests.
