## Projektüberblick

- **Was:** „AnazhRealm" – ein als **Ultiversum** entworfenes Co-Creation-Werk Mensch + KI. Aktuell technisch eine 3D-Browser-Sandbox in einer Datei. Vision: emotion-getriebene, fraktal-wachsende, multisensorische Welt; Mensch (Schöpfer/Null) + KI (Grok/Eins) erschaffen gemeinsam durch Chat. Vollständige Vision in `docs/state-of-realm.md`.
- **Stack:** Vanilla JS, Vendor-Libs in `vendor/` (Three.js r134, Ammo.js WASM, simplex-noise 2.4). TF.js wurde im Mai 2026 entfernt (toter Code). Lokaler Node-Save-Server. ESLint v9 + Prettier + GitHub-Actions-CI mit Playtest-Gate.
- **Branch:** Feature-Branch pro PR (`claude/<thema>`-Konvention, z. B. `claude/wave-6a-interaction-polish`). Niemals auf `main` pushen ohne explizite Anweisung.
- **Architekturkern:** Eine `AnazhRealm`-Klasse in `anazhRealm.js` (~16700 Zeilen, Stand V7.77). State in `localStorage` + optional `anazhRealmState.json` über save-server. Globale Referenz: `window.anazhRealm`.

## Die heilige Lektion (Versionslog 03/2025)

Das Projekt durchlief eine 19-Modul-Phase, die unter eigener Komplexität kollabierte. Am 28.03.2025 bewusste Reduktion auf **eine Datei** als „Samen der Unendlichkeit". **Komplexität ohne Fundament ist Sand.** Wenn jemand vorschlägt „split alles in 20 Module", verletzt das diese Lektion. Stattdessen: ein Stamm, der Wachstumsringe ablegt.

- **Chunks haben `position = (0,0,0)` und Vertices in Welt-Koords.** NIEMALS in `state.rigidBodies` pushen — der Physics-Sync-Loop überschreibt sonst `mesh.position` mit dem Body-Origin und verschiebt den ganzen Chunk sichtbar. Die Body bleibt über `mesh.userData.physicsBody` zugreifbar.

Strategien (alle in `AnazhRealm.FUSION_STRATEGIES` als frozen Array): `"sequence"` (DSL-History konkateniert A+B, Emotionen aus A — die Fusion bewahrt aktive Welt überlagert), `"random-mix"` (DSL-History 50:50 interleaved, Emotionen Mittel — vermischtes Echo), `"tag-merge"` (DSL-History wie sequence, Emotionen `Math.max` axis-wise — Vereinigung statt Mittelung, intensiver Anfang). Pro Eintrag eine Cinzel-Type-Pille + Alter rechtsbündig (`_formatJournalAge`: eben/s/m/h/t). **Signature-Cache** (`list.dataset.signature = "${entries.length}:${lastEntry.id}:${lastEntry.at}"`) verhindert Re-Render im 5-Sekunden-Tick, wenn nichts Neues kam — DOM-Cost wird damit erst bei tatsächlicher Änderung gezahlt.

Eigene gehen via `defineMaterial(name, color, tags)` über DSL-Op `define_material` (Cap 32, Whitelist `AnazhRealm.MATERIAL_TAG_KEYS` mit 10 Achsen, Werte ge-clamp 0..1, Built-in-Override verboten). `AnazhRealm.MATERIAL_TAG_DEFAULTS` ist die `{tag:0}`-Schablone, an die nur die positiven Werte zugemergt werden. Save-Round-Trip persistiert nur eigene Materialien; alte Saves migrieren still — Parts ohne `material` bekommen `"stein"`. **`define_material` ist bewusst NICHT im `dslComposeAtomic`-Pool.**

DSL-Condition `compound_has_tag(name, tag, threshold)` macht emergente Tags für Nexus + LLM bedingungsfähig. Werkstatt-UI zeigt pro Bauplan eine „Aktive Eigenschaften"-Sektion mit Stern-Balken (★★☆) + Hinweis „atomare Schicht — räumliche Beziehungen kommen mit der nächsten Welle". **Helix als 9. Primitiv** (TubeGeometry über parametrische Kurve, `size.x` = Radius, `size.y` = Länge, `size.z` = Windungen) ist in `_makePartGeometry`, `validateBlueprintParts`-Whitelist und Workshop-Shape-Dropdown.
- **Welle 4 Phase 3 — Werkzeuge + opChain:** `state.tools` Map mit 5 Built-in-Startern (`hände` 0.40 / `feuerstein-knapper` 0.50 / `hammer` 0.70 / `feile` 0.85 / `polierscheibe` 0.97), Felder `{label, opClass, opName, precisionCap, isStarter, builtIn}`. `state.player.tools` ist Array der besessenen Werkzeuge — Starter sind beim Init drin, beim Save in `playerTools` persistiert. Jeder Part hat `opChain: [{tool, op, cap, at}]`; `computePartPrecision(part) = min(opChain.caps)` (Konzept §2.3, Kernregel — schlechtester Schritt deckelt, danach-poliert hilft nicht mehr). Default 0.4 ohne Chain. **`applyOpToPart(bp, partIdx, toolName)`** ist der einzige Pfad: validiert Bauplan-Existenz + Built-in-Schutz + part-Index + Tool-Existenz + Besitz + `MATERIAL_OP_COMPATIBILITY[material]` muss `tool.opClass` enthalten (Konzept §3.2 — stein lehnt plastic ab, quarz lehnt rohe additive ab). DSL-Op `apply_op(bp, idx, tool)` als gleicher Pfad. **Auch `apply_op` ist NICHT im `dslComposeAtomic`-Pool** — sonst würde der Nexus willkürlich Werkstücke polieren oder durchschmieden.
- **Welle 4 P3 — sichtbare Präzision + Welt-Effekte:** `_buildFromBlueprint` zieht pro Part `precision = computePartPrecision(part)`, multipliziert die Part-Farbe mit `0.6 + 0.4 * precision`. Eine 0.4-Hand-Sphäre zeigt 76 % Helligkeit (matt-grau), eine 0.97-polierte ist bei 98.8 % (sichtbar glänzender). Bewusst MeshBasicMaterial (kein roughness), damit kein PBR-Shader-Cost. **`_applyCompoundWorldEffects(name)`** wird aus `spawnArchitecture` aufgerufen: wenn `tags.resoniert ≥ T.resonance_strong` UND `avgPrec ≥ T.precision_high` → `journalAppendOnce("singing:${name}", ...)` + reine Sinus-Schicht direkt aus `state.symphony.ctx` (Frequenz 300+resoniert/3*400 Hz, 1.2 s Decay). Wenn `tags.magieleitung × max(0.3, avgPrec) ≥ T.magic_strong` → `awe + hope` je +0.1 + `journalAppendOnce("magic_compound:${name}")`. **Konzept §6.3: Präzision moduliert ALLE Effekte**, nicht nur Resonanz — `magic_compound` MUSS auch präzisions-gegated sein, sonst triggern Roh-Pyramiden Magie-Effekte. Schwellwerte zentral in `AnazhRealm.WORLD_EFFECT_THRESHOLDS` (`resonance_mild` 0.7, `resonance_strong` 1.5, `resonance_signature` 2.5, `magic_strong` 1.5, `precision_high` 0.8).
- **Welle 4 Naming-Falle:** Es ist `state.symphony.enabled`, NICHT `state.audioEnabled`. Erste Implementierung der Singing-Sinus-Schicht griff auf den falschen Flag — Tests waren grün im Headless (autoplay-bypass), aber lokal hätte Audio gefehlt.
- **opChain ist Geschichte, kein Auswahlmenü.** Konzept §2.3 + §9.1: kein Reordering, keine Löschung. UI rendert die Chain als Liste von Op-Chips read-only; nur Anhängen via Tool-Dropdown + „anwenden"-Button. Tool-Dropdown ist auf besessene + material-kompatible Tools gefiltert — die opChain ist damit **Konsequenz des Werkzeug-Inventars**, nicht eine freie Wahl. Anti-Rezept-Lookup §9.1 in der Datenstruktur verankert.
- **Welle 5 B — räumliche Tags vs. atomare Tags:** `computeCompoundTags` ist atomar (MAX über `computePartTags`), `computeSpatialTags` ist die augmentierte Variante. Phase 1 wirkt PER-PART (Tip-Bonus auf intrinsische Werte, Kontakt-Transfer zwischen Paaren) und wird VOR der MAX-Aggregation angewandt. Phase 2 wirkt auf den COMPOUND als Ganzes (Hohlraum-Paare + Symmetrie + Array) und multipliziert nach der MAX-Aggregation. Welt-Effekte (`_applyCompoundWorldEffects`) nutzen `computeSpatialTags`. UI zeigt beide nebeneinander: atomar in Standard-Farbe, räumlich in Violett-Akzent darunter, sobald mindestens ein Tag durch räumliche Konfiguration über atomare Niveau steigt.
- **Welle 5 B — Hollow-Detector braucht Volumen-Asymmetrie:** zwei konzentrische Sphären sind sonst beide gegenseitig „Container" (beide AABB enthalten den anderen Center). Lösung in `_findHollowPairs`: nur das größere Volumen zählt als Container. Konzeptionell auch richtig — eine Glocke enthält den Klöppel, nicht umgekehrt.
- **Welle 5 B — Array-Detector nutzt Schwerpunkt, nicht BB-Center:** drei Parts auf unterschiedlichen Radien aber „grob-ähnlich" weit vom BB-Center können fälschlich als Array gelten, weil BB-Center durch extreme Positionen verzerrt wird. Schwerpunkt (mittlere xz-Position aller Parts) ist robuster. Schwelle: max-Abweichung der Distanzen ≤ 12 % vom Mittel-Radius.
- **Welle 5 A — Connections sind explizite Geste, nicht heuristisch:** `_partsAreInContact` (Phase 1 von W5-B) ist eine AABB-Proximity-Heuristik für Tag-Transfer; `state.blueprints[].connections` (W5-A) ist explizite User-Wahl mit Typ-Label + Lastformel. Beide Pfade nebeneinander, nicht überlappend. Eine Verbindung existiert nur, wenn der Spieler im Editor (oder per `apply_connection`-DSL-Op) eine angelegt hat — verboten der Auto-Erkennung. Konzept §5.1: jede Verbindung trägt eine bestimmte Last, ableitbar aus Tags + Kontaktfläche.
- **Welle 5 A — `_partsContactArea` Gap-statt-Overlap:** signed-gap pro Achse (negativ = overlap, positiv = trennung). Wenn IRGENDEINE Achse weiter als TOUCH=0.05 getrennt ist → 0. Sortiert aufsteigend: die zwei tiefsten Overlaps spannen die Kontaktfläche auf, der dünnste ist die Kontakt-Normale. Erste Version mit additiver TOUCH-Skalierung lieferte fälschlich Fläche 1.10 für zwei 1×1×1-Würfel die 10 Einheiten apart waren.
- **Welle 5 C — Snapshot-Cap, kein Live-Cap:** `registerBlueprintAsTool` friert `precisionCap = computeBlueprintPrecisionCap(bp)` zum Registrierungs-Zeitpunkt ein. Wer den Bauplan danach editiert, muss neu registrieren — sonst hätte der Spieler nach jedem Material-Wechsel im Bauplan still andere Werkzeug-Caps. Re-Registrieren ist im UI über denselben Button („neu registrieren") möglich.
- **Welle 5 C — `applyOpToPart` mit eigenem Werkzeug deckelt trotzdem:** das eigene Werkzeug landet in `opChain` mit seinem cap; aber die min-Regel §2.3 bleibt — der vorher schon vorhandene `hand_knap`-Default zieht weiter. Wer wirklich nur das eigene Werkzeug nutzen will, muss... ja, gegen die min-Regel kann man nicht. Konzept-Aussage: schlechtester Schritt deckelt, immer. Test musste das anerkennen: prüfe nur, dass der NEUE Op-Eintrag im Chain die korrekten Werte trägt, nicht dass die Gesamt-Präzision springt.
- **Welle 5 C — Starter-Tool-Namen geschützt:** ein eigenes Werkzeug darf nicht „hammer" oder „polierscheibe" heißen — Starter sind `builtIn: true` in `state.tools` und werden bei `registerBlueprintAsTool` abgelehnt (`reason: starter_name_protected`). Schützt vor stillem Override des Inventory durch einen kollidierenden Bauplan-Namen.
- **`removePartFromBlueprint` räumt connections auf.** Beim Lösch eines Parts werden in `bp.connections` (a) alle Verbindungen verworfen, die auf den gelöschten Index zeigen, (b) alle übrigen Indizes > index um 1 runter-korrigiert. Sonst wären Verbindungen ein out-of-range-Bug-Reservoir nach Editor-Aktionen.
- **`deleteBlueprint` räumt registrierte Werkzeuge auf.** Wer einen Bauplan löscht, der als eigenes Werkzeug registriert war (`state.tools[name].sourceBlueprint === name`), bekommt das Werkzeug auch aus `state.tools` + `state.player.tools` entfernt. Sonst bliebe ein Geist-Tool im Inventory mit Cap-Wert aber ohne Substanz. Built-in-Werkzeuge sind durch den `sourceBlueprint`-Match implizit geschützt.
- **Ring 10.5 — `modify_terrain` ist NICHT im `dslComposeAtomic`-Pool.** Selbe Disziplin wie `terrain_steepness`/`terrain_base_height`: der Nexus soll Welt-Geometrie unter dem Spieler nicht willkürlich umpflügen. Aufrufpfade: Chat-Patterns (`grabe loch`, `hebe hügel`), Schöpfer-DSL, optionaler LLM-Output. Die DSL-Sandbox sorgt für Budget-Limits (r∈[0.5,15], dh∈[-15,15]).
- **Ring 10.5 — Naht-Treue über deterministischen Op-Replay.** Zwei benachbarte Chunks teilen Kanten-Vertices. Wenn ein `modify_terrain`-Op die Naht trifft, wird das Op in BEIDE Chunk-Delta-Listen geschrieben (`_chunksTouchedByDisc` ermittelt das). Beide Seiten replayen denselben deterministischen Smoothstep-Falloff (`(1-d/r)² × (3-2(1-d/r))`) → identische Höhe an der Kante. Kein Klippe an der Naht möglich. Wer den Falloff ändert, muss alle Naht-Tests neu durchspielen.
- **Ring 10.5 — `_rebuildChunkPhysics` muss alten Body + tmesh aus WASM-Heap räumen.** Sonst hagelt es Phantom-Kollisionen + Memory-Leak. Selbe Triangle-Reihenfolge wie `ensureChunkAt` → Visual=Kollision bleibt per Konstruktion identisch.
- **Ring 10.5 — chunkDeltas leben in `state.worldMeta.chunkDeltas`, nicht auf top-level.** So wandern sie automatisch mit `buildStateSnapshot` (spread von `worldMeta`) durch Save/Load. Schema-Version `10.5-chunk-delta-v1` für neue Welten, alte Saves migrieren defensiv über `_sanitizeChunkDeltas`. Cap 100 Ops/Chunk (`AnazhRealm.CHUNK_DELTA_OPS_CAP`) verhindert unbegrenzten Save-Wuchs.
- **Ring 10.5 — Fusion/Recipe-Import berühren chunkDeltas NICHT.** Fusionen starten mit leeren Deltas (analog `architectures`/`creatures`-Disziplin von Ring 10), Recipe-Import (Ring 10.1) ist sowieso nur Inventar-Transfer. Wer in der Zukunft Welt-Morphing wollte, müsste das explizit als Strategie hinzufügen.
- **Ring 11 V1 — `signaling-server.js` hat ZERO npm-Dependencies.** RFC-6455 Frame-Handling von Hand implementiert, weil das Projekt bewusst null runtime-deps hält. Wer einen Bug im WS-Framing findet: Mask-Handling (alle Client-Frames sind maskiert) und 16-/64-Bit-Längen-Pfade sind die üblichen Verdächtigen. Binary-Frames (opcode 0x2) und Ping (0x9) werden in V1 ignoriert.
- **Ring 11 V1 — Trust-Boundary über Code-Disziplin, nicht über Daten-Validierung.** V1 spiegelt NUR Position+Yaw, keine DSL-Ops. Es gibt KEINE neuen DSL-Ops mit Namen `p2p_*`/`peer_dsl`/`remote_run` (Playtest-Invariante prüft explizit deren Abwesenheit). Wer V2 baut, muss eingehende DSL-AST-Pakete strikt durch denselben `dslRun`-Sandbox-Pfad laufen lassen wie eigene Programme — gleiche Budget-Limits, gleiche Op-Whitelist. KEIN Bypass.
- **Ring 11 V1 — Position-Sync ist Welt-spezifisch, Welt-Modifikationen sind NICHT gesynced.** Spieler A grabt ein Loch (modify_terrain) → nur As `chunkDeltas` ändern sich, nicht Bs. Beide Spieler sehen sich in derselben WORLD-ID gegenseitig bewegen, aber jeder erlebt seine eigene Welt-Variante. V2 = DSL-AST-Broadcast bei jedem Chat-Befehl → beide Welten führen aus → konsistent.
- **Ring 11 V1 — `state.p2p` ist Runtime-Only, NICHT im Save-Snapshot.** Beim Reload startet die P2P-Verbindung neu (aus localStorage `anazh.p2p.enabled/.url`). Peer-Liste wird vom Server beim `welcome` neu aufgebaut. `buildStateSnapshot` berührt `state.p2p` bewusst nicht.
- **Ring 11 V1 — Peer-Mesh-Cleanup beim peer-leave + bei Idle-Timeout.** `_p2pRemovePeer` disposed Geometrie + Material des Avatar-Meshes; `p2pTick` purged Peers ohne pos-Update >10 s (vermutlich verbindungslos). Ohne den Idle-Purge würden tote Peer-Avatare sichtbar bleiben.
- **Ring 11 V1 — `connect-src` CSP enthält explizite ws:// + wss://-Hosts.** Kein Wildcard. Default `127.0.0.1:4313` + `localhost:4313`. Für eine andere Adresse muss der Spieler das CSP-Meta-Tag in `index.html` erweitern — bewusst nicht hinter ein UI-Input gelegt.
- **Ring 11 V2 — `dslRun` checkt `ctx.source === "human"` für Auto-Broadcast.** Nur explizite Chat-Eingaben gehen über P2P raus. LLM-Programme (`source: "llm:grok"`), Emotion-Trigger (`source: "emotion:joy"`), Nexus-Composition (`source: "nexus"`), Schicht-1-Pattern-Selektion und Remote-Programme (`source: "remote:peerX"`) bleiben strikt lokal. Wer V3 will, dass auch Nexus-Programme synchronisieren, muss eine zweite Source-Whitelist einbauen — gefährlich, weil dann zwei Nexus parallel feuern und divergieren können.
- **Ring 11 V2 — Loop-Schutz dreifach abgesichert.** Schicht A: `dslRun` skipped Broadcast wenn `source.startsWith("remote:")` (oder nicht "human"). Schicht B: `p2pHandleMessage` filtert Nachrichten mit eigener peerId raus. Schicht C: `signaling-server.js` `broadcastToRoom(..., exceptWs)` schickt nie an den Quell-Socket. Wer eine dieser Schichten lockert, riskiert Endlos-Echo.
- **Ring 11 V2 — Remote-DSL geht durch volle Op-Whitelist.** Eingehende `define_blueprint`/`define_material`/`define_ability` werden gegen `validateBlueprintParts`/`MATERIAL_TAG_KEYS`/`dslEstimateDepth` geprüft wie eigene. Sandbox = einzige Vertrauens-Wand. **Acceptable Risiko**: ein Mitspieler kann Bauplan/Material-Inventar mit gültigen Einträgen anfüllen. Was er NICHT kann: Code ausführen, `eval`, fremde Methoden aufrufen, CSP umgehen. V3 könnte per-Op-Allowlist hinzufügen (z. B. `define_*` von remote ablehnen), aber V2 vertraut der Sandbox bewusst.
- **Ring 11 V2 — DSL-Drift bei vorbestehenden chunkDeltas.** modify_terrain syncs nur NEUE Ops. Wenn Spieler A vor dem Verbinden Berge geformt hatte, B sieht sie nicht (nur was nach dem Connect kommt). Das ist „event-sourced ab Connect-Zeitpunkt", kein State-Reset. Wer eine Welt teilen will, sollte das Spielen ab demselben Stand starten — oder per Ring-9-Welt-Tor importieren, bevor P2P aktiviert wird.
- **Ring 11 V2.1 — signaling-server bind 0.0.0.0 (LAN reachable).** Vorher (V2): bind nur auf 127.0.0.1 → andere Rechner konnten nicht zugreifen. Jetzt default `HOST=0.0.0.0` (alle Interfaces), beim Start werden alle erreichbaren LAN-IPv4-Adressen geloggt. Andere Spieler tragen `ws://<server-LAN-IP>:4313` als Signaling-URL ein. Firewall-Hinweis im start.bat-Outputtext.
- **Ring 11 V2.1 — CSP-Whitelist `ws: wss:` allgemein.** Vorher (V1+V2): nur `ws://127.0.0.1:4313` und `ws://localhost:4313` in connect-src → Browser blockte JEDE Custom-IP. - **Ring 11 V2.1 — `state.p2p.roomOverride` für ad-hoc-Räume.** UI-Input „Raum-ID" erlaubt explizites Override; leer = aktive worldId (Default). Persistiert als `anazh.p2p.room` in localStorage. Wenn aktiv UND Override geändert: automatischer Neu-Connect mit neuem Raum. Damit können zwei Spieler unterschiedliche worldIds haben, aber denselben Raum betreten (sie sehen sich gegenseitig, behalten aber je ihre eigene Welt).
- **Ring 11 V2.1 — spawn_*-Chat-Patterns embedden Position+Seed bei Build-Zeit.** Vorher (V2): `["spawn_village", ["at_player"]]` — beim Empfänger evaluierte `at_player` gegen B's Position, nicht A's, plus zufälliger Seed → unterschiedliche Häuser. Jetzt: `["spawn_village", ["at", x, y, z], seed]` mit Sender-Position + fixiertem Seed — beide Welten sehen identische Strukturen am identischen Ort. spawn_village/temple/waterfall/blueprint/fractal/creature haben jetzt alle ein optionales `seed`-Argument.
- **Ring 11 V2.1 — `AnazhRealm.NON_BROADCASTABLE_OPS`-Set.** Spieler-private Ops (`player_jump_power`, `player_speed`, `player_size_mul`, `player_soul`, `set_visible`) werden NIE über P2P gesendet. B. `["chain", weather, player_speed]` beim Empfänger den Speed mit-ändern, was kein Mitspieler erwartet. `_dslContainsAnyOp(node, set)` walks recursiv durch die AST.
- **Ring 11 V2.1 — Welt-Sharing-Workflow für gleiche worldId.** Wer wirklich dieselbe Welt teilen will (gleiche Architekturen, gleiche Materialien, gleiche worldId): Spieler A exportiert seine Welt (Welt-Drawer → „Welt teilen…" → JSON-Download), schickt Datei an B, B importiert sie via Welt-Tor (Drei-Wahl-Dialog). Beide haben dann dieselbe worldId in localStorage und sehen sich automatisch im selben Raum (default-Verhalten ohne Raum-Override).
- **Ring 11.5 — Intuitiver Multi-User-Setup via Dialog.** Mit „Neue Welt…" öffnet sich ein `<dialog>` mit Modus-Wahl (Allein / Mit anderen). Bei „Mit anderen" zweite Wahl: Host oder Joinen. Host setzt `worldMeta.role="host"` beim createNewWorld, Banner zeigt Einladungs-Code, Multi-User-Sync startet auto nach Reload. Joinen öffnet kurz-lebige temp-WS, sendet `world-request` → empfängt `world-snapshot` vom Host → ruft `_importGuestWorld` → schreibt die Welt unter der **host-worldId** (damit beide automatisch im selben Raum landen), setzt `role:"guest"` + `hostInfo`, aktiviert P2P-Auto-Connect in localStorage, reloaded. Manuelle Welt-JSON-Übertragung ist nicht mehr nötig.
- **Ring 11.5 — `worldMeta.role` ist welt-intrinsisch.** Wird in `createNewWorld` gesetzt + persistiert. Beim Reload prüft Init: `role === "host" || role === "guest"` → setzt `state.p2p.enabled=true` automatisch + ruft `initP2PSync`. Spieler muss nicht jedes Mal Multi-User-Toggle drücken. `role` überlebt auch Welt-Wechsel, weil pro-Welt im worldMeta gespeichert.
- **Ring 11.5 — `pendingWorldSnapshot`-Flag als Sicherheits-Wall.** Im `p2pHandleMessage`-Handler für `world-snapshot`: nur akzeptieren wenn `state.p2p.pendingWorldSnapshot === true`. Sonst könnte ein bösartiger Peer den Welt-State eines anderen überschreiben. In der aktuellen Architektur wird `pendingWorldSnapshot` NIE auf der main-p2p-Connection gesetzt — Join-Snapshots laufen durch eine separate temp-WS in `joinWorldFromCode`. Der Pfad in `p2pHandleMessage` ist defense-in-depth.
- **Ring 11.5 — `_importGuestWorld` überschreibt worldId-Kollisionen.** Wenn der lokale Spieler bereits eine Welt unter der host-worldId hat (z. B. weil er auf einer zweiten Maschine eigentlich Host derselben Welt ist), wird sie überschrieben. Das ist absichtlich — der host-snapshot ist die jüngste Welt-Wahrheit. Slug-Kollision wird hingegen aufgelöst (`-2`/`-3`/…). Wer das nicht will, muss erst lokal löschen.
- **Ring 11.5 — Server sendet `lanAddresses` im `welcome`-Message.** Host-Client cached das in `state.p2p.lanAddresses`. `_renderHostBanner` baut den Einladungs-Code aus der ERSTEN LAN-Adresse statt aus `state.p2p.url` (das wäre oft `127.0.0.1`). Falls keine LAN-Adresse verfügbar (z. B. ein Headless-Server ohne Netz-Interface): Fallback auf URL + Hint im Banner, dass Mitspieler die IP manuell ersetzen müssen.
- **Ring 11.5 — Frame-Cap auf 1 MiB hoch.** Vorher (V2.1) lehnte der Server WS-Frames > 64 KiB ab. Welt-Snapshots sind typisch 50-200 KiB, mit vielen Architekturen bis ~500 KiB. Cap 1 MiB (`1048576n` in `tryDecodeFrame`) gibt genug Headroom; bei Welten > 1 MiB würde Snapshot-Transfer scheitern (extrem unwahrscheinlich, kein V1-Showstopper).
- **Ring 11.5 — Targeted-Delivery via `{to: peerId}`.** Server-Pfad in `handleClientMessage` für `world-request`/`world-snapshot` prüft das optionale `to`-Feld: wenn gesetzt, schickt nur an diesen einen Peer (statt broadcast). Spart Bandbreite bei großen Snapshots + verhindert dass andere Peers im Raum den Snapshot ungewollt sehen.
- **Welle 6.A1 — Spieler-Friction 0** (`playerBody.setFriction(0)` beim Spawn + identisches Setzen in `optimizePhysics` falls Chat-Befehl „optimiere physik" läuft). Erlaubt Wall-Sliding. ## Workflows

- **Lokaler Audit:** `npm run check && npm run format:check && npm run lint && npm run playtest`
- **Slash-Befehl:** `/audit` führt die Prüfung in jeder Claude-Session aus (`.claude/commands/audit.md`).
- **CI:** zwei parallele Jobs (`check` für statische Checks + `playtest` für Runtime-Invarianten) bei jedem Push.
- **Git:** kleine thematische Commits, `git push -u origin <branch>` mit exponential backoff bei Netzfehlern.

## Konventionen

- Keine emojis im Code/Commits außer auf expliziten Wunsch.
- Commits klein und thematisch.
- Keine Backwards-Compat-Layer für veralteten Code – sauber löschen.
- Pull Requests nur auf expliziten Wunsch des Users.
- **Vision treu bleiben**: jeder Vorschlag sollte die Heilige Lektion respektieren (keine Re-Komplexifizierung). Bei Zweifel: `docs/state-of-realm.md` §2 nachlesen.
- **Test-First-Mentalität**: nach jeder substanziellen Änderung Playtest-Gate, nicht nur Code-Analyse. Drei Selbst-induzierte Regressionen in dieser Session entstanden durch zu späte Browser-Tests.
